Search Typeahead System — Project Report

HLD101 Assignment · Author: Parth Dagia · Repository: github.com/parthdagia05/HLS-ASSIGN

A backend-focused search typeahead (autocomplete) system, similar to the suggestion box in Google / Amazon / YouTube. It suggests popular queries as the user types, records searches, serves suggestions from a distributed cache (Redis nodes routed with consistent hashing), ranks results with a recency-aware "trending" score, and absorbs write pressure with batched writes.

2.65 mssuggestion latency (server p95, warm)
96%cache hit rate
≈50×write reduction via batching
~1/Nkeys remapped on node loss

Tech stack: Python 3.12 + FastAPI · PostgreSQL 16 (primary store) · Redis (3 logical cache nodes, app-level consistent hashing) · plain HTML/CSS/JS frontend · Docker Compose for local infrastructure.

1. Architecture

Two independent paths. The read path (suggestions) is optimised for latency — cache first, database only on a miss. The write path (search submissions) is optimised for throughput — buffer in memory, aggregate, and write to the database in batches.

Data model

A single table keeps the system easy to reason about. The index idx_queries_prefix ON queries (query text_pattern_ops) makes WHERE query LIKE 'prefix%' use an index even on a UTF-8 database, which keeps prefix lookups fast at 120k rows.

ColumnMeaning
query (PK)search string, stored normalised (lowercased, trimmed)
countall-time number of searches — historical popularity
recent_scoretime-decayed recent activity — drives trending
recent_score_updated_atwhen recent_score was last refreshed (decay anchor)
last_searched_atlast submission time (diagnostics)

Concurrency model

Intentionally synchronous and simple: FastAPI runs sync endpoints in a threadpool; PostgreSQL access goes through a connection pool; the batch writer runs on its own daemon thread with lock-guarded shared state. No asyncio, so there are no await subtleties to explain.

2. Dataset — source & loading

The system needs query,count rows (≥ 100,000 unique queries).

  • Default (synthetic, reproducible): scripts/generate_dataset.py builds 120,000 unique queries from real-world head terms ("iphone", "java"…) + modifiers ("review", "near me", "2024"…), with a Zipf-like (long-tail) count distribution — the shape real search traffic has. It is deterministic (fixed seed), so the dataset is identical on every run and needs no download. Output: data/queries.csv.
  • Using a real dataset instead: produce a CSV with a query,count header (any open dataset of search queries / product names / page titles; aggregate to derive counts if absent) and run python scripts/load_dataset.py --csv path/to/your.csv.

Loading: load_dataset.py bulk-loads via PostgreSQL COPY into a staging temp table, then does a single INSERT … ON CONFLICT DO UPDATE into queries — fast, and idempotent (safe to re-run). Every query is normalised on load to match how the API normalises the user's prefix.

python scripts/generate_dataset.py        # writes data/queries.csv (120k rows)
python scripts/load_dataset.py            # COPY → staging → upsert into queries

4. Design choices & trade-offs

Why PostgreSQL + an indexed prefix scan (not a trie / search engine)?

The dataset (100k–1M rows) is small enough that an indexed LIKE 'prefix%' returns the top 10 in single-digit milliseconds, and a relational table makes counts, recency and batched upserts trivial and durable. The cache absorbs the read volume, so the database is rarely the bottleneck. A trie or Elasticsearch would be the next step at much larger scale.

Why our own consistent hashing (not Redis Cluster)?

With plain hash(key) % N, adding/removing a node changes the bucket for almost every key, dumping the entire load onto the database at once. Consistent hashing places nodes and keys on a circle; a key belongs to the next node clockwise, so a node change only moves ~1/N of keys. Each physical node is placed at many points (virtual nodes, 150) so load is even. Doing it in app code keeps the interesting logic visible and explainable. Trade-off: we manage routing ourselves instead of getting it free from Redis Cluster — accepted, because the assignment is about understanding the mechanism.

Trending — recency-aware ranking (the 5 required points)

  1. How recent searches are tracked: each row has recent_score; on every search we decay the stored score by elapsed time then add the new activity: recent_score = recent_score·e^(−λ·Δt) + delta, with λ = ln2 / half_life (default 10 min).
  2. How recency affects ranking: trending score = count + α · decayed_recent_score (α scales recent activity to be comparable to all-time counts), so a recently-hot query gets a temporary boost.
  3. Avoiding permanent over-ranking of a brief spike: the score decays exponentially — once the burst stops, the boost halves every half-life and the query slides back to its count-based position.
  4. Cache update/invalidation when rankings change: a changed query invalidates its prefixes' cached entries in both modes; short TTL is a backstop. Basic and trending use separate cache keys.
  5. Trade-offs (freshness vs latency vs complexity): shorter TTL / aggressive invalidation = fresher but more misses; the decay is computed in SQL at read time, keeping writes simple at a tiny per-read cost. Defaults: 30s TTL, per-flush invalidation, 10-min half-life — all single config values.

Batch writes & failure trade-offs

Writing to Postgres on every search is wasteful. Instead submissions are buffered and aggregated (repeated queries summed), then flushed in a single multi-row upsert either every FLUSH_INTERVAL_SECONDS or once BATCH_SIZE distinct queries are buffered. Result: ≈50× fewer DB writes.

What if it crashes before a flush? The buffer is in memory, so we add an optional write-ahead log: each submission is appended to a file; on flush we atomically rename the WAL aside, write the batch, then delete it; on startup we replay any leftover WAL. Honest limits: we flush to the OS but do not fsync per write (covers a process crash, not a power crash — a throughput trade-off), and recovery is at-least-once (a crash in the tiny window after commit but before deleting the WAL replays that batch — slight over-count). With the WAL disabled the writer is pure in-memory: faster, but a crash loses the un-flushed buffer.

Measured locally (12-core Linux, Docker Postgres + 3 Docker Redis nodes, 120k-query dataset). Reproducible via scripts/benchmark.py, batch_demo.py, cache_demo.py.
